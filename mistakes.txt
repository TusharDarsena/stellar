Here are the major bugs and architectural disconnects you need to patch before this goes live:

### 1. The Quantity Illusion (`PurchasePage.tsx`)
[cite_start]The UI allows an attendee to increment the ticket quantity (up to 10) and calculates a multiplied `totalPrice`[cite: 1904, 1927, 1930]. However, the blockchain integration completely ignores this. 
* [cite_start]**The Bug:** The `handlePurchase` function only generates a single `ticketId` and calls `purchaseTicket` once[cite: 1910, 1911]. 
* **The Impact:** A user who clicks "+" to buy 5 tickets will see the total price for 5 tickets, but the transaction will only mint and charge them for **1 ticket**. [cite_start]The Soroban contract's `purchase` signature inherently only handles one ticket per call[cite: 1368, 1746]. You need to either disable the quantity selector for the MVP or loop the `purchaseTicket` call (which will trigger multiple Freighter prompts, so disabling it is safer for now).

### 2. Scanner Freighter Spam (`ScannerPage.tsx`)
[cite_start]The `ScannerPage` relies on `Html5Qrcode`, which continuously fires its success callback as long as a readable QR code is in the camera frame[cite: 1969, 1970].
* [cite_start]**The Bug:** `handleScan` does not pause the camera or debounce the scan event before executing `await markUsed(...)`[cite: 1966]. 
* **The Impact:** Holding the camera over a valid QR code will fire `markUsed` multiple times a second. [cite_start]Because the organizer is using Freighter, this will spawn dozens of overlapping signature request popups, effectively freezing the organizer's wallet or browser[cite: 1696]. You must pause `html5QrCode` the moment a valid payload is decoded, wait for the transaction to complete, and then offer a manual "Scan Next" button to resume.

### 3. Organizer Funds are Locked Forever (`DashboardPage.tsx`)
[cite_start]The MVP relies on a pull-based escrow model where organizers release their funds after the event date[cite: 1392]. 
* [cite_start]**The Bug:** The "Release Funds" button calls `handleRelease`, but this function is just a stub containing `console.log(Release funds for ${eventId})`[cite: 2085]. 
* [cite_start]**The Impact:** The actual `releaseFunds` transaction builder exported from `src/lib/soroban.ts` is never imported or executed in this file[cite: 2077, 2078]. Organizers literally cannot withdraw their XLM.

### 4. The Void of Metadata (`CreateEventPage.tsx` & `useEvents.ts`)
[cite_start]Because you chose not to use an off-chain database for the MVP (D-004), event metadata relies entirely on the contract storage[cite: 1395].
* [cite_start]**The Bug:** The `CreateEventPage` form asks organizers to painstakingly fill out a `venue`, `city`, `imageUrl`, and `description`[cite: 2038, 2040, 2050, 2052]. [cite_start]However, the `createEvent` API call to Soroban only passes `organizer`, `eventId`, `name`, `dateUnix`, `capacityXlm`, and `priceStroops`[cite: 1750]. 
* **The Impact:** All that rich metadata is instantly dropped and never saved anywhere. [cite_start]Consequently, `useEvents.ts` is forced to hydrate every single event fetched from the RPC with the exact same hardcoded fallback image, description, venue, and city[cite: 1658, 1659]. To fix this without a database, you either need to add a generic `metadata_uri` string to the Soroban `Event` struct to hold an IPFS/JSON link, or remove those fields from the UI to avoid misleading organizers.

### 5. Infinite "Active" Tickets & Fake History (`MyTicketsPage.tsx`)
[cite_start]The user's ticket library is broken into "ACTIVE" and "HISTORY" tabs[cite: 1885, 1886].
* [cite_start]**The Bug:** The active tab maps over the `tickets` array without verifying the status[cite: 1888, 1890]. [cite_start]Furthermore, the "HISTORY" tab is populated entirely by hardcoded HTML mock data (e.g., *Stellar Gala 2024*)[cite: 1895]. 
* [cite_start]**The Impact:** Once an attendee's ticket is scanned at the door and marked `'Used'` on-chain, it will still sit in their "ACTIVE" tab forever[cite: 1788, 1888]. You need to filter `tickets.filter(t => t.status === 'Active')` for the first tab, and `tickets.filter(t => t.status !== 'Active')` for the history tab.

### 6. Unforgiving QR Clock Drift (`qr.ts` & `QRDisplayPage.tsx`)
[cite_start]The architecture opted for an absolute timestamp check instead of a windowed floor (D-006)[cite: 1399]. 
* **The Bug:** The payload expires strictly when `|now - timestamp| >[cite_start]= 30` seconds[cite: 1725]. [cite_start]Simultaneously, the `QRDisplayPage` sets a 30-second visual countdown and regenerates the payload exactly every 30 seconds[cite: 1938, 1941].
* **The Impact:** If the attendee's phone clock is ahead of the organizer's phone clock by even 1.5 seconds, a freshly generated QR code could instantly evaluate as expired the millisecond it is scanned. You need to introduce a slight grace period (e.g., checking `|now - timestamp| <= 45`) in the scanner logic to account for standard device clock drift while maintaining the 30-second UI rotation.
