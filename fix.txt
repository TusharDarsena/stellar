2. Bug Analysis: Organizer Can't Create Event
Root Cause
The Frontend Bindings Are Out of Sync

Contract Spec (Rust)	Frontend Bindings (TypeScript)
name parameter	String (type 16)	Symbol (type 17)
The WASM spec encoded in the frontend bindings defines name as Symbol, but the deployed Rust contract expects String.

Symbol (type 17): Only allows a-z, A-Z, 0-9, _ — any other byte fails
String (type 16): Allows arbitrary UTF-8 bytes
The error "byte 25 not allowed in Symbol" (0x19 = EM control character) comes from the SDK rejecting characters that aren't valid Symbol bytes.

Why This Happened
When the contract was modified (from an earlier version), the name field in Rust changed from Symbol to String, but the frontend bindings were never regenerated from the updated WASM.

## Recommended Schema

### 1. `events` — Event metadata (extends on-chain)

```sql
CREATE TABLE events (
  -- Primary key (links to on-chain)
  event_id TEXT PRIMARY KEY,
  organizer_address TEXT NOT NULL,
  
  -- Rich metadata (NOT on-chain)
  name TEXT NOT NULL,
  description TEXT,
  image_url TEXT,
  venue TEXT,
  city TEXT,
  category TEXT,
  tags TEXT[],
  
  -- Mirror for easy read (synced via contract events)
  date_unix BIGINT,
  capacity BIGINT,
  price_per_ticket BIGINT,
  current_supply BIGINT DEFAULT 0,
  status TEXT DEFAULT 'Active',
  
  -- Timestamps
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_events_status ON events(status);
CREATE INDEX idx_events_date ON events(date_unix);
CREATE INDEX idx_events_organizer ON events(organizer_address);
```

### 2. `organizers` — Organizer profiles

```sql
CREATE TABLE organizers (
  address TEXT PRIMARY KEY,
  name TEXT,
  logo TEXT,
  description TEXT,
  website TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 3. `profiles` — User profiles

```sql
CREATE TABLE profiles (
  address TEXT PRIMARY KEY,
  display_name TEXT,
  avatar_url TEXT,
  is_organizer BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

# Honest Recommendation

Based on reading architecture.md and AGENTS.md, here's my honest analysis:

## Current Architecture

| User Type | Wallet | Sign-In | Profile |
|----------|--------|--------|---------|
| **Organizer** | Freighter | ✅ (Freighter handles) | ❌ |
| **Attendee** | Burner Wallet | ❌ (D-028) | ❌ |

> D-028 says: Attendee uses Burner Wallet — Web3Auth **deferred to post-MVP**

---

## The Problem

Currently, attendees have:
- Random keypair generated in-browser
- Secret stored in localStorage
- No profile, no identity
- On page reload → new keypair (session lost) because Zustand is in-memory only

---

## Recommended Sync Strategy: Contract-First with Edge Function (Option C)

### Why NOT Options A or B:

| Option | Problem |
|--------|--------|
| **A**: Supabase first | If contract tx fails, Supabase has orphan data |
| **B**: Contract first, then Supabase in `.then()` | If Supabase write fails, data inconsistency |

### Recommended: Supabase Edge Function

```
┌──────────────┐     ┌──────────────┐
│  Frontend    │────▶│  Supabase    │
│              │     │  Edge Fn     │
└──────────────┘     └──────────────┘
       │                     │
       │ validates          ▼
       │             ┌──────────────┐
       └─────────────▶│  Soroban     │
                     │  Contract   │
                     └──────────────┘
```

**Why**:
1. Single write point — always consistent
2. Edge Function handles both contract call + Supabase write
3. No frontend double-write logic
4. Can retry from Edge Function if needed

---

## For Attendee Sign-In

Given this is a hackathon and you want simplicity, I recommend:

| Option | Pros | Cons |
|--------|------|------|
| **Burner + Supabase profile** | Keep current wallet, add profile | No real "sign in" |
| **Supabase Auth (email/password)** | Real sign-in, profiles | Extra complexity |
| **Supabase Auth + social** | Same + Google login | Requires OAuth setup |

### My Honest Recommendation: **Start Simple**

**Keep burner wallet** for blockchain operations. Add **Supabase profiles** for identity:

```sql
-- profiles table (already recommending)
CREATE TABLE profiles (
  address TEXT PRIMARY KEY,     -- Burner wallet address (auto-linked)
  display_name TEXT,
  avatar_url TEXT,
  email TEXT,              -- For account recovery, not required
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

- Attendee keeps using burner wallet for transactions
- Profile is auto-created on first "connect wallet" based on address
- Option to add email later

This gives you attendee identity without the complexity of adding Supabase Auth in MVP.

# Fix Plan: Attendee Wallet Connection Failure

## Issue
Attendees cannot connect their wallet. Clicking "Connect Wallet" results in a 400 Bad Request from friendbot.stellar.org with the error:
```
"detail": "account already funded to starting balance"
```

Organizer wallet connection (via Freighter) works fine.

## Root Cause
In `frontend/src/lib/stellar.ts`, the `fundWithFriendbot()` function handles the case where an account already exists by checking for the string `createAccountAlreadyExist` in the friendbot error response:

```typescript
if (res.status === 400 && body.includes('createAccountAlreadyExist')) return;
```

However, the actual error message returned by friendbot is:
```
"account already funded to starting balance"
```

Since this does not contain `createAccountAlreadyExist`, the check fails and the function throws an error. This error propagates up to `connectAttendee()` in `useWallet.ts`, which catches it and logs it, but never completes the wallet connection.

## Files Involved
- `frontend/src/lib/stellar.ts` — Contains `fundWithFriendbot()` with the buggy check
- `frontend/src/hooks/useWallet.ts` — Calls `fundWithFriendbot()` in `connectAttendee()`
- `frontend/src/App.tsx` — Routes to `connectAttendee()` for attendee views

## Fix
Update the condition in `frontend/src/lib/stellar.ts` to also catch the `"already funded"` error message:

```typescript
// BEFORE:
if (res.status === 400 && body.includes('createAccountAlreadyExist')) return;

// AFTER:
if (res.status === 400 && (body.includes('createAccountAlreadyExist') || body.includes('already funded'))) return;
